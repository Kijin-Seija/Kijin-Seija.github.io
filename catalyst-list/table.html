# 列表数据渲染

# 基础配置

对于一个列表页来说，数据的展示是它最基础的部分，下面我们基于测试数据来生成一个列表。

mockdata = [
  { "entity_id": 1, "title": "河原的偶像水子", "name": "戎璎花" },
  { "entity_id": 2, "title": "古代鱼的携子护卫", "name": "牛崎润美" },
  { "entity_id": 3, "title": "地狱口岸的守护神", "name": "庭渡久侘歌" },
  { "entity_id": 4, "title": "鬼杰组组长", "name": "吉吊八千慧" },
  { "entity_id": 5, "title": "埴轮兵长", "name": "杖刀偶磨弓" },
  { "entity_id": 6, "title": "孤立无援造就的造形神", "name": "埴安神袿姬" },
  { "entity_id": 7, "title": "劲牙组组长", "name": "骊驹早鬼" },
]

根据示例数据，我们可以配置如下的columns:

'columns': [
  {
    'label': 'ID'
    'field': {
      'type': 'raw',
      'key_name': 'entity_id',
    },
  },
  {
    'label': '称号',
    'field': {
      'type': 'raw',
      'key_name': 'title',
    },
  },
  {
    'label': '姓名'
    'field': {
      'type': 'raw',
      'key_name': 'name',
    },
  }
]

以下是常用的配置参数：

  • label 列名称，作为该列头部的名称显示

  • width 固定列的宽度，在自适应宽度不能满足需求时使用

  • field 列内容配置，在该条目下的所有配置都会作为该列内容渲染的相关参数

  • field.type（必填） 列内容类型，此处我们使用的所有类型是raw - 原始类型，使用该类型意味着我们将数据中对应项原封不动的展示在表格中。

  • field.key_name （必填） 列对应数据的key，在我们的示例中，可以看到每个key_name都对应着json数据中的一项，我们会将对应项的内容作为该列的数据。

column配置完成后，我们需要配置数据来源，在本例中因为我们使用的是测试数据，所以以下的配置只是形式上的，但实际中数据一般都是和后端接口交互得到的，因此以下的配置也是必须的、

首先，我们需要在actions中配置请求数据的uri

'actions': {
  'search': {
    'uri': '/api/mock/teacher/list',
  }
},

然后，我们需要实现一个provider，$mpSeacch函数，该函数接收我们配置的uri，使用者基于自己项目中的ajax方案进行请求发送和接收。该函数为一个async函数，需返回一个如下格式的对象或者resolve该对象

$mpSearch: async (uri) => {
    // 因为是测试数据，所以这里没有使用uri
    return Promise.resolve({
      list: this.mockdata,
      total: 100,
      currentPage: 1
    })
  }
};

  • list 当前页数据，list中每个数据实体必须定义entity_id字段，作为唯一标识，在本例中，list的内容就是我们的测试数据。

  • total 数据条目总数，用于分页

  • currentPage 当前页数，从1开始计数

完成以上配置之后，我们就可以显示我们的测试数据了。

# 分页

分页的配置非常简单，只需要在$mpSearch的用户实现请求方法中传递页数即可。$mpSearch拥有第二个参数params，包含了分页字段：page_size（每页数据条目数量）、page（页码）。 示例如下（假设我们使用的是axios）：

provide() {
  return {
    $mpSearch: async (url, params) => {
      return axios.get(url, {params: {
        page: params.page
        limit: params.page_size
      }})
    }
  };
}

查看demo

注意

  • page页码从1开始计数。
  • 当你改变了每页显示条数设置后，所有应用该组件的地方每页条数都会发生变化，并且该变化被缓存至localStorage，这并不是bug

# 处理枚举型数据

有时候，我们的数据长下面这样：

mockdata = [
  { "entity_id": 1, "title": "河原的偶像水子", "name": "戎璎花", "fromArtical": 17 },
  { "entity_id": 2, "title": "古代鱼的携子护卫", "name": "牛崎润美", "fromArtical": 17 },
  { "entity_id": 3, "title": "地狱口岸的守护神", "name": "庭渡久侘歌", "fromArtical": 17 },
  { "entity_id": 4, "title": "鬼杰组组长", "name": "吉吊八千慧", "fromArtical": 17 },
  { "entity_id": 8, "title": "接近神的蝴蝶妖精", "name": "爱塔妮缇拉尔瓦", "fromArtical": 16 },
  { "entity_id": 9, "title": "跨越浮世门关的山姥", "name": "坂田合欢", "fromArtical": 16 },
  { "entity_id": 10, "title": "醉心神佛的守护神兽", "name": "高丽野阿吽", "fromArtical": 16 },
  { "entity_id": 11, "title": "垂迹于森林的魔法地藏", "name": "矢田寺成美", "fromArtical": 16 },
],

fromArticalType不是直接用来显示的数据，通常我们有一个key-value键值对来告知不同的type对应的内容：

articalName = [
  ...
  {value: 16: label: '东方天空璋'},
  {value: 17: label: '东方鬼形兽'},
  ...
]

使用enum_text，我们可以定义这种转换。

columns: [
  ...
  {
    'label': '来自作品',
    'field': {
      'type': 'enum_text',
      'key_name': 'fromArtical',
      'options_id': 'articalNames'
    },
  },
  ...
]

options_id定义了一个options键值对的数据来源, 要获取正确的数据，我们需要在组件中实现另一个provider $mpGetOptions

$mpGetOptions是一个async函数，它接受optionsId作为参数，使用者根据optionsId去获取对应的options键值对。可以本地配置，也可以从后台获取然后返回一个Promise

$mpGetOptions: async (optionsId) => {
  if (optionsId === 'articalNames') return [
    {value: 16, label: '东方天空璋'},
    {value: 17, label: '东方鬼形兽'},
  ]
}

注意

option的格式固定为{value, label}

不止能枚举数据，catalyst-list还提供了枚举图标的组件，类型名为enum-icon

图标的尺寸可以通过width参数控制

columns: [
  ...
  {
    'label': '作品图标',
    'field': {
      'type': 'enum_icon',
      'width': '40px',
      'key_name': 'fromArtical',
      'options_id': 'articalIcons'
    },
  },
  ...
]

$mpGetOptions: async (optionsId) => {
  if (optionsId === 'articalNames') return [
    {value: 16, label: '东方天空璋'},
    {value: 17, label: '东方鬼形兽'},
  ]
  if (optionsId === 'articalIcons') return [
    {value: 16, label: 'https://16.jpg'},
    {value: 17, label: 'https://17.jpg'},
  ]
}

根据上面的配置而成的完整效果可以查看demo

# 处理格式化的数据类型

catalyst-list目前支持对时间点，时间段，金钱的处理。

# 时间点

处理时间点的组件为datetime, 在这里可以额外定义一个可选参数format，规定时间的显示格式。如果不配置，默认的格式为yyyy-MM-dd HH:mm:ss(2019-03-05:13:20:45)。 相关的模板字符串见这里

columns: [
  ...
  {
    'label': '时间点',
    'field': {
      'type': 'datetime',
      'key_name': 'time',
      'format': 'yyyy-mm-dd'
    },
  },
  ...
]

# 时间段

处理时间段的组件类型为timespan, 同样可以定义format显示格式。默认的格式为h小时m分ss秒 相关的模板字符串见这里

columns: [
  ...
  {
    'label': '时间段',
    'field': {
      'type': 'timespan',
      'key_name': 'duration',
      'format': 'hh:mm:ss'
    },
  },
  ...
]

# 金钱

处理金钱的字段的组件类型为money，会将传入的数值自动加上人民币符号和逗号分隔。

columns: [
  ...
  {
    'label': '价格',
    'field': {
      'type': 'money',
      'key_name': 'price'
    },
  },
  ...
]

# 可交互的内容

有些时候，我们希望表格项可以点击，以跳转到我们设定好的页面，或者是执行一些操作。交互组件interaction可以满足这样的需求。

例如，在之前的数据中，我们希望点击姓名可以跳转到对应的内容，我们可以将之前姓名部分的配置修改成如下形式:

columns: [
  ...
  {
    'label': '姓名'
    'field': {
      'type':'interaction',
      'key_name': 'name',
      'mode':'page',
      'uri':'/detail',
    }
  }
  ...
]

在mode: page下，该列表项的entity_id将作为uri的query一并传过来，实际访问的地址将会是/detail?entity_id=1

我们还可以不关联数据配置，只生成固定的文本。这需要在数据中配置label

columns: [
  ...
  {
    'label': '操作'
    'field': {
      'type':'interaction',
      'label': '查看',
      'mode':'page',
      'uri':'/detail',
    }
  }
  ...
]

实际上，interaction类型是通过action组件实现的，当我们介绍用户操作配置时，你会发现这些配置都是一致的，action里的所有配置都可以被应用到这里。

# 复合类型

当列表项需要由多种上述类型拼装而成时，我们需要使用另一个参数fields fields接受一个数组。数组中的每一项就是上述类型的field配置对象。

比如，我们新增一栏『操作』，其中拥有两个按钮，『查看』和『编辑』。我们的配置如下：

columns: [
  ...
  {
    'label': '操作',
    'fields': [
      {
        'type':'interaction',
        'label': '查看',
        'mode':'page',
        'uri':'/detail'
      },
      {
        'type':'interaction',
        'label': '编辑',
        'mode':'page',
        'uri':'/edit'
      }
    ]
  },
  ...
]

复合类型fields数组里还能接受普通的字符串和html文本，并将其渲染在指定位置上

columns: [
  ...
  {
    'label': '操作',
    'fields': [
      '修饰字1'
      {
        'type':'interaction',
        'label': '查看',
        'mode':'page',
        'uri':'/detail'
      },
      '<p style="color: red;">修饰字2</p>'
      {
        'type':'interaction',
        'label': '编辑',
        'mode':'page',
        'uri':'/edit'
      }
    ]
  },
  ...
]

在复合类型的fields里，除了可以渲染上述基础类型的组件外，我们还有一种特殊的组件类型divider，它的配置如下:













 
 
 
 











columns: [
  ...
  {
    'label': '操作',
    'fields': [
      '修饰字1'
      {
        'type':'interaction',
        'label': '查看',
        'mode':'page',
        'uri':'/detail'
      },
      {
        type: 'divider'
        text: '<p style="color: #999"> | </p>'
      }
      {
        'type':'interaction',
        'label': '编辑',
        'mode':'page',
        'uri':'/edit'
      }
    ]
  },
  ...
]

在之后我们会提及到的权限管理中，fields中的每个元素都会基于权限决定它是否展示。如此一来，如果使用一个固定字符串作为分隔符，当元素的显示状态发生变化时，分隔符有可能被渲染在开头或结尾，这样的分隔符是无效的。而divider会自动判断它的前后是否有元素，如果前面或后面没有渲染出元素，那么它将会自动隐藏。

# 排序

只需要我们在column配置里设置sortable: true，便可调用排序功能, 此时list组件接受一个sort-change事件，该事件的触发机制与参数和element table 相同（见这里）, 除此之外，事件的参数里额外传递了一项field，为当前排序列的field配置

例：

columns: [
  ...
  {
    'label': '价格',
    'sortable': true,
    'field': {
      'type': 'money',
      'key_name': 'price'
    }
  },
  ...
]

我们也可以对复合类型进行排序。要注意的是，复合类型只会基于其中第一个被设置为sortable: true的元素进行排序。

# 多级表头

在任意类型的column配置里，我们可以定义chirdren字段，该字段接受一个数组，数组中再定义内部的column配置，这样就定义好了一个多级表头。

注意，如此定义以后，外层的column就只做表头使用，field(s)不再有效。

columns: [
  ...
  {
    'label': '来自作品',
    'children': [
      {
        'label': '作品名称',
        'field': {
          'type': 'enum_text',
          'key_name': 'fromArtical',
          'options_id': 'articalNames'
        },
      },
      {
        'label': '作品图标',
        'field': {
          'type': 'enum_icon',
          'width': '40px',
          'key_name': 'fromArtical',
          'options_id': 'articalIcons'
        },
      },
    ]
  }
  ...
]
